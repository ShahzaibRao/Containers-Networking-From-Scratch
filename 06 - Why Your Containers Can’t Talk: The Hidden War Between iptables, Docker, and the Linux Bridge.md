# Why Your Containers Can’t Talk: The Hidden War Between iptables, Docker, and the Linux Bridge

Docker networking is not magic. It is Linux bridge + veth pairs + Netfilter (iptables).

When containers suddenly stop communicating, it is usually not Docker itself.
It is the interaction between:

* Linux bridge (Layer 2)
* Netfilter hooks (Layer 3)
* The `FORWARD` chain policy
* The `bridge-nf-call-iptables` kernel setting

This guide builds everything manually so you can see exactly what is happening.

---
![Docker Impact on Linux iptables](sources/assets/images/Docker%20Impact%20onLinux%20iptables.png)

# The Gatekeeper of the Linux Kernel

Netfilter is the packet filtering framework inside the Linux kernel.
iptables is just the user interface to manage those rules.

When a packet travels through the system, it hits different chains:

| Chain   | Purpose                          |
| ------- | -------------------------------- |
| INPUT   | Traffic destined for the host    |
| OUTPUT  | Traffic generated by the host    |
| FORWARD | Traffic passing through the host |

Important detail:

When `net.bridge.bridge-nf-call-iptables=1`,
even pure bridge traffic is forced through the `FORWARD` chain.

That is where most production issues start.

# Lab Setup (No Docker, Pure Linux)

We will create:

* Two namespaces: `ns1`, `ns2`
* One bridge: `br0`
* VETH pairs
* Same subnet: `10.22.33.0/24`

Goal:
Make ping work, then break it using iptables.

---

# Step 1: Create Namespaces

```bash
$ sudo ip netns add ns1
$ sudo ip netns add ns2
```

Verify:

```bash
$ ip netns list
ns1
ns2
```

---

# Step 2: Create Bridge

```bash
$ sudo ip link add br0 type bridge
$ sudo ip addr add 10.22.33.1/24 dev br0
$ sudo ip link set br0 up
```

Verify:

```bash
$ ip -br addr show br0
br0             UP             10.22.33.1/24
```

---

# Step 3: Create VETH Pairs

```bash
$ sudo ip link add veth1 type veth peer name veth1-br
$ sudo ip link add veth2 type veth peer name veth2-br
```

Attach bridge sides:

```bash
$ sudo ip link set veth1-br master br0
$ sudo ip link set veth2-br master br0
$ sudo ip link set veth1-br up
$ sudo ip link set veth2-br up
```

Move peers to namespaces:

```bash
$ sudo ip link set veth1 netns ns1
$ sudo ip link set veth2 netns ns2
```

---

# Step 4: Assign IP Addresses

Inside ns1:

```bash
$ sudo ip netns exec ns1 ip addr add 10.22.33.2/24 dev veth1
$ sudo ip netns exec ns1 ip link set veth1 up
$ sudo ip netns exec ns1 ip link set lo up
```

Inside ns2:

```bash
$ sudo ip netns exec ns2 ip addr add 10.22.33.3/24 dev veth2
$ sudo ip netns exec ns2 ip link set veth2 up
$ sudo ip netns exec ns2 ip link set lo up
```

Verify:

```bash
$ sudo ip netns exec ns1 ip -br addr
veth1           UP             10.22.33.2/24
```

```bash
$ sudo ip netns exec ns2 ip -br addr
veth2           UP             10.22.33.3/24
```

---

# Step 5: Test Connectivity (Works)

```bash
$ sudo ip netns exec ns1 ping 10.22.33.3
64 bytes from 10.22.33.3: icmp_seq=1 ttl=64 time=0.07 ms
```

Bridge is working as pure Layer 2 switch.

No routing involved.

---

# Step 6: Break the Network

Set default `FORWARD` policy to DROP:

```bash
$ sudo iptables -P FORWARD DROP
```

Check:

```bash
$ sudo iptables -L FORWARD -n -v
Chain FORWARD (policy DROP)
```

Now try ping again:

```bash
$ sudo ip netns exec ns1 ping 10.22.33.3
```

It fails.

Even though:

* Same subnet
* Same bridge
* No routing required

Why?

Because bridged traffic is being passed to iptables.

---

# The Hidden Switch: bridge-nf-call-iptables

Check current value:

```bash
$ sysctl net.bridge.bridge-nf-call-iptables
net.bridge.bridge-nf-call-iptables = 1
```

When set to 1:

Bridge traffic enters the `FORWARD` chain.

Disable it:

```bash
$ sudo sysctl -w net.bridge.bridge-nf-call-iptables=0
net.bridge.bridge-nf-call-iptables = 0
```

Try ping again:

```bash
$ sudo ip netns exec ns1 ping 10.22.33.3
64 bytes from 10.22.33.3: icmp_seq=1 ttl=64 time=0.06 ms
```

It works again.

Now the bridge behaves like pure Layer 2.

---

# Why Docker Requires bridge-nf-call-iptables=1

Docker needs to:

* Apply NAT rules
* Enforce inter-container isolation
* Control published ports

Without bridge → iptables integration:

Containers could bypass firewall rules.

So in Docker environments:

```bash
net.bridge.bridge-nf-call-iptables = 1
```

is required.

That means your `FORWARD` chain must be configured properly.

---

# The Correct Secure Fix

Instead of disabling bridge filtering, allow bridge traffic explicitly:

```bash
$ sudo iptables -I FORWARD 1 -i br0 -o br0 -j ACCEPT
```

Verify:

```bash
$ sudo iptables -L FORWARD -n -v
```

Now ping works even with policy DROP.

This is the secure approach.

---

# Understanding conntrack

Netfilter uses connection tracking.

Common rule:

```bash
$ sudo iptables -A FORWARD -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
```

What it does:

* First packet must match an allow rule
* Subsequent packets automatically allowed
* Reduces rule complexity

Always place this rule at the top for performance.

# Docker Chain Injection

If Docker is installed, run:

```bash
$ sudo iptables -L -n
```

You may see:

* DOCKER
* DOCKER-USER
* DOCKER-ISOLATION

Important:

Docker modifies `FORWARD` automatically.

Best practice:

Add custom rules in:

```bash
DOCKER-USER
```

Docker will not overwrite that chain.

Example:

```bash
$ sudo iptables -I DOCKER-USER -i br0 -o br0 -j ACCEPT
```

# Real Production Failure: Silenced Microservice

Scenario:

Security team sets:

```bash
iptables -P FORWARD DROP
```

Result:

* Containers cannot talk to each other
* Host can still reach containers
* Containers can reach host
* Only container-to-container traffic is broken

Check packet counters:

```bash
$ sudo iptables -vnL FORWARD
```

You will see DROP counter increasing.

The fix is targeted ACCEPT rules — not reverting policy to ACCEPT.

---

# Performance Insight

iptables is processed linearly.

If your rule order is:

1. Rare rule
2. Rare rule
3. Rare rule
4. Most common rule

Every packet evaluates first three rules unnecessarily.

Best practice:

1. ESTABLISHED,RELATED
2. Most common traffic
3. Less frequent rules
4. Default DROP

---

# Advanced Optimization: raw Table and NOTRACK

High traffic systems may overload conntrack.

Check usage:

```bash
$ cat /proc/sys/net/netfilter/nf_conntrack_count
$ cat /proc/sys/net/netfilter/nf_conntrack_max
```

If full → packets dropped silently.

To bypass tracking for specific traffic:

```bash
$ sudo iptables -t raw -A PREROUTING -p udp --dport 53 -j NOTRACK
```

Use carefully.
